Quick reference

Five Git words

01

Repository

The project folder plus its complete change history.

02

Commit

A named snapshot saved on your computer.

03

Pull

Download Benson’s newest work before you start.

04

Push

Upload your commits. A push to master publishes the site.

05

Deploy

Make a version live for players.

master is the live lane. Treat every push like a production release. Never force-push.

01

Account setup

Create your accounts

  1. Open github.com/signup ↗.
  2. Choose GitHub Free, a username you like, and a unique password.
  3. Open GitHub’s email and verify your address.
  4. Send Benson your exact GitHub username. Don’t create a repository.
  1. On GitHub, open Settings → Password and authentication.
  2. Enable two-factor authentication with an authenticator app or passkey.
  3. Save the recovery codes in your password manager.

Never send Benson your password, authenticator codes, or recovery codes.

  1. Open Cloudflare sign-up ↗.
  2. Create and verify the account.
  3. Send Benson the account email. Don’t create a Worker.

Prepare Windows

Open Claude Code and paste this prompt. Claude can inspect the computer, install anything missing, and walk you through the browser sign-in. Approve each action only after reading it.

I already have Claude Code working on this Windows computer. Please prepare it
for collaborating on Lastlight.

Check whether Git, GitHub CLI, Node.js LTS, npm, and Python 3.13 are installed.
Install only what is missing, using normal Windows tools. Then help me sign in
to GitHub with my own account and configure Git commits with my real name and
GitHub noreply email.

Pause whenever I need to complete a browser sign-in or approve something.
Never ask me to paste a password, two-factor code, recovery code, or access
token into this chat. Explain what you are changing, and finish by verifying
that every tool and the GitHub login work.

Change, check, and ship

Start Claude Code inside the repository. Replace the bracketed line with the change you want, then paste the whole prompt. Claude will handle the terminal work, tests, commit, push, and deployment checks.

I am Jacob, working under Benson's standing authorization to make this
Lastlight change directly on master. Work only inside lastlight/**.

Task: [DESCRIBE THE CHANGE AND WHAT “DONE” LOOKS LIKE]

Before editing, confirm the branch is master, the tree is clean, and
origin/master is pulled with --ff-only. Implement and test the change, run
the complete Lastlight GitHub Actions check suite, and review the final diff.

Never force-push, discard work, or touch other project folders. Stop on a
conflict, a check you cannot fix, or an unexpected file outside lastlight/**.

When everything is green, make one focused commit, synchronize with the newest
origin/master, push directly to master, watch Lastlight Checks and GitHub Pages,
and verify the live game. If lastlight/worker/** changed, test and deploy the
Cloudflare relay separately, report its version, and verify /health.

Before approving the push: read Claude’s summary. Confirm it changed only Lastlight files and that every local check passed.

Recover safely

Paste the exact error or describe what went wrong in the bracketed line. Claude should diagnose it without erasing work or rewriting shared history.

Something went wrong while I was working on Lastlight:

[PASTE THE EXACT ERROR AND DESCRIBE THE LAST THING THAT HAPPENED]

Diagnose the current state before changing anything. Preserve all uncommitted
work and explain the safest recovery plan in plain language.

Never force-push, use git reset --hard, delete unfamiliar files, expose secrets,
or touch anything outside lastlight/**. If a bad website commit is already live,
use a new revert commit rather than rewriting history. If a bad relay deployment
is live, show me the rollback plan and ask before executing it. Rerun the full
Lastlight checks after any fix and verify production afterward.

The two never-commands: never force-push shared master, and never use git reset --hard to solve a collaboration problem.
